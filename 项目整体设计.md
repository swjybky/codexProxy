# Codex Proxy Desktop 项目整体设计

## 1. 产品边界

该应用不是通用 AI 网关，也不承担用户、渠道、计费或数据库管理。它只为单机用户提供一条稳定的 Codex 转发链路：

```text
Codex CLI / OpenAI Responses Client
        │  Bearer 本地 API Key
        ▼
127.0.0.1:1455/v1/responses[/compact]
        │  请求字段适配 + OAuth Header
        ▼
chatgpt.com/backend-api/codex/responses[/compact]
        │
        └── JSON / SSE 原样返回
```

## 2. 技术分层

| 层 | 文件 | 职责 |
|---|---|---|
| 桌面壳 | `app/main.py` | 启动本地服务与 pywebview 窗口，退出时关闭服务 |
| HTTP 入口 | `app/server.py` | 静态资源、管理 API、本地鉴权、Responses 路由、SSE 输出 |
| Codex 适配 | `app/codex_proxy.py` | 请求转换、上游 URL、Codex Header、OAuth 刷新 |
| 本地存储 | `app/storage.py` | 设置、API Key、OAuth 凭证、原子写入与 JWT 字段提取 |
| 路径 | `app/runtime_paths.py` | 源码/冻结资源路径与跨平台用户数据目录 |
| 管理界面 | `web/src/` | React + TypeScript + Vite 单页桌面 UI |

## 3. 与 new-api Codex Adaptor 对齐

普通 Responses 请求：

1. 只保留 `OpenAIResponsesRequest` DTO 中定义的字段。
2. 按配置注入或合并 `instructions`；缺失时补空字符串。
3. 强制 `store=false`。
4. 删除 `max_output_tokens` 与 `temperature`。
5. 转发到 `/backend-api/codex/responses`。

Compact 请求：

1. 只保留 `model`、`input`、`instructions`、`previous_response_id`。
2. 注入或补齐 `instructions`，不执行普通 Responses 的字段改写。
3. 转发到 `/backend-api/codex/responses/compact`。

两类请求统一设置：

```text
Authorization: Bearer <access_token>
chatgpt-account-id: <account_id>
OpenAI-Beta: responses=experimental
originator: codex_cli_rs
Content-Type: application/json
Accept: application/json | text/event-stream
```

## 4. 本地 API

对客户端：

- `POST /v1/responses`
- `POST /v1/responses/compact`
- `GET /v1/models`（仅供客户端模型发现，不产生上游请求）

这三个接口要求 `Authorization: Bearer <本地 API Key>`。其他 Chat Completions、Embeddings、Images 等接口均不实现。

对桌面界面：

- `GET /api/status`
- `PUT /api/settings`
- `POST /api/credentials/import`
- `POST /api/credentials/import-default`
- `POST /api/credentials/refresh`
- `POST /api/key/regenerate`
- `POST /api/connection-test`

## 5. 安全约束

- HTTP 服务硬绑定 `127.0.0.1`，设置页不能改为外网监听。
- 上游 OAuth Token 与本地 API Key 分离，客户端永远接触不到上游 Token。
- 桌面管理令牌与客户端 API Key 再次隔离；令牌通过 URL fragment 注入并立即转存 `sessionStorage`，不会进入 HTTP 请求行或日志。
- 所有 `/api/*` 管理接口验证 `X-Admin-Token`，阻止其他网页发起本机 CSRF 操作。
- 凭证原子写入用户数据目录，权限为 `0600`。
- 状态接口只返回账号摘要，不返回任何 Token。
- 日志只记录请求路径与 HTTP 状态，不记录请求体和鉴权 Header。
- 上游测试必须由用户点击触发，因为它会产生一次真实 Responses 请求。
- 请求前检查凭证有效期，距离过期不足五分钟时自动刷新；上游首次返回 401 时仅刷新并重试一次，避免无限重试。

## 6. 数据文件

`settings.json` 保存上游地址、网络代理、系统提示词、默认模型和本地 API Key；`credentials.json` 保存标准化后的 OAuth 字段。应用不依赖数据库。

## 7. 打包

前端先构建为 `web/dist/`，PyInstaller 再将 `app/`、静态资源与应用图标打进产物。Windows 使用 `.ico` 与 Edge WebView2；macOS 生成带 `.icns` 的 `.app`；Linux onedir 产物附带透明 PNG、freedesktop 启动器与 `hicolor` 图标资源。运行时 pywebview 也会显式加载同一 PNG，确保 Linux Qt/GTK 窗口与任务栏显示一致。
